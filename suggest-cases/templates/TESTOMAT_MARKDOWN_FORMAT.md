## Test Case Fields

### Priority Values

| Value | Use For |
|-------|---------|
| `critical` | Must not fail â€” core business flows, payment, auth |
| `high` | Important functionality â€” key features, main workflows |
| `normal` | Standard functionality â€” regular features (default) |
| `low` | Nice-to-have â€” cosmetic, edge cases, exploratory |

### Tags

Tags use `@tag_name` format (lowercase, kebab-case):

**Testing type tags:**
- `@smoke` â€” smoke test
- `@regression` â€” regression test
- `@e2e` â€” end-to-end test
- `@api` â€” API test
- `@ui` â€” UI test
- `@unit` â€” unit test
- `@integration` â€” integration test
- `@security` â€” security test
- `@performance` â€” performance test
- `@accessibility` â€” accessibility test
- `@exploratory` â€” exploratory test
- `@visual` â€” visual regression test
- `@mobile` â€” mobile test
- `@contract` â€” contract test

**Scenario type tags:**
- `@happy-path` â€” standard successful flow
- `@negative` â€” invalid input, error scenarios
- `@boundary` â€” boundary value tests
- `@edge-case` â€” edge case scenarios
- `@destructive` â€” destructive/adversarial tests

**Test design tags:**
- `@test-design` â€” generated by test design techniques
- `@equivalence-partitioning` â€” equivalence partitioning technique
- `@boundary-value` â€” boundary value analysis
- `@decision-table` â€” decision table technique
- `@state-transition` â€” state transition testing
- `@pairwise` â€” pairwise/combinatorial testing
- `@data-driven` â€” data-driven test
- `@risk-based` â€” risk-based test selection
- `@generated-from-code` â€” generated from source code analysis

**Feature/module tags:**
- `@auth` â€” authentication
- `@orders` â€” order management
- `@payments` â€” payment processing
- `@users` â€” user management
- `@checkout` â€” checkout flow
- Use custom tags per project: `@[module-name]`

**Source/process tags:**
- `@charter` â€” exploratory testing charter
- `@tour` â€” exploratory testing tour
- `@flaky` â€” known flaky test
- `@suggested` â€” AI-suggested test case
- `@found-by-exploratory` â€” converted from exploratory finding

### Labels

Labels must be existing project labels in Testomat.io. Common labels:
- `Automated` â€” test has automated implementation
- `Manual` â€” manual test only
- `Needs Review` â€” requires review before finalization
- `Deprecated` â€” scheduled for removal

### Test IDs

- Format: `@Txxxxxxxx` (e.g., `@T123ff599`)
**Try to use existing one** form the tms.

## Section Headers

Use these section headers within test cases:

| Section | Purpose | Required |
|---------|---------|----------|
| `### Requirements` | Links to business requirements | Optional |
| `### Pre-conditions` | Setup needed before test execution | Optional |
| `### Steps` | Numbered actions with expected results | Required |

### Steps Format

Each step is a numbered action followed by an expected result:

```markdown
### Steps
* Action description
    *Expected Result*: What should happen...

* Next action description
    *Expected Result*: What should happen...
```

**Rules:**
- Each step should have a corresponding `*Expected Result*`
- Actions should be specific and reproducible
- Expected results should be verifiable (not vague)
- Include specific data values where relevant

## Rules Summary

1. **Suite** = file, **Tests** = sections within a file
2. `<!-- test -->` header comment is required before each test
3. `<!-- suite -->` header comment is optional for suite metadata
4. Priority values: `low`, `normal`, `high`, `important`, `critical`
5. Tags use `@tag_name` format â€” can be in header or in test title
6. Labels must be existing labels from the project
7. Test IDs (`@Txxxxx`) are never added manually â€” created during sync
8. Each step should have a corresponding `**Expected Result**`
9. Use `### Steps`, `### Pre-conditions`, `### Requirements` as section headers
10. File names use **kebab-case**: `login-flow.md`, `order-lifecycle.md`
11. Directory structure organizes suites by area/feature
